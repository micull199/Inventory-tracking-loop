# Changelog

High-level "what shipped" log. One line per slice. Newest at the top.

## Unreleased

- **C1** ‚Äî Per-item `requires_checkout` flag UI + checkouts schema foundation: new `Checkout` ORM model in `app/models.py` mirroring MISSION ¬ß6 columns (no status enum ‚Äî open vs returned derived from `returned_at IS NULL`). Migration `0013_create_checkouts.py` creates the table + four indexes (per item / item_unit / user / returned_at). Items list gains a "Checkout?" column showing "Yes" / "‚Äî" plus a new filter nav (`All items` / `Requires checkout`) backed by `?requires_checkout=yes` (silent coerce on any other value). The items form gains an explanatory help note under the existing checkbox. No new routes ‚Äî check-out / check-in routes land in C2 / C3, manager "currently out / overdue" view in C4. C1's job is to land the foundation so those slices don't have to also ship a migration.
- **PO4** ‚Äî Email PO PDF to supplier (draft ‚Üí sent): new `POST /admin/purchase-orders/{po_id}/send` (Manager + Office). Validates draft + active supplier with non-blank email, renders the PDF via `app.pdf.render_po_pdf`, builds an `EmailMessage` with HTML body + PDF attachment, delivers via the configured backend, then flips `po.status = SENT` + `po.sent_at = now()` and writes a `purchase_order.sent` audit row. New `app/email_backend.py` module exposes `EmailMessage` / `EmailAttachment` / `EmailBackend` Protocol + `ConsoleEmailBackend` (in-memory outbox + stdout summary; default in dev/tests) and `SmtpEmailBackend` (stdlib `smtplib` + `email.message`). Switch via the `EMAIL_BACKEND` env var. Detail page gains `po-send-form` on drafts with supplier email, or `po-send-blocked-no-email` otherwise.
- **PO2** ‚Äî Draft PO from a supplier group: new module `app/purchase_orders.py` with three routes ‚Äî `POST /admin/reorder/draft-po` (Manager + Office; creates a draft PO from a supplier group on the reorder dashboard), `GET /admin/purchase-orders` (list, status-filterable), `GET /admin/purchase-orders/{po_id}` (read-only detail). New tables `purchase_orders` + `purchase_order_lines` (migration `0011`); new `POStatus` enum (only `draft` written in PO2). New `nav-pos` link in `base.html` for admin/manager/office. The reorder dashboard gains a "Draft PO" button per active-supplier group; archived-supplier and "(no supplier)" groups render distinct blocked-notes. Lines auto-default `qty_ordered` from `item.reorder_qty` (or deficit, or 1) and `expected_unit_cost` from the most recent cost layer (or null). Audit: `purchase_order.created` carrying the full lines snapshot. Editing per-line fields and cancelling drafts is deferred to PO2b ‚Äî without it MISSION ¬ß3's "expected unit cost is editable" requirement is not yet satisfied, so DoD #6 cannot tick.
- **PO1** ‚Äî Reorder dashboard: new module `app/reorder.py` + `GET /admin/reorder` for **Manager + Office** (read-only; Workshop + pending blocked). Surfaces items where `current_qty <= reorder_threshold AND archived_at IS NULL`, grouped by supplier (with a synthetic "(no supplier)" bucket and an "(archived)" suffix on archived suppliers). Each row shows current qty, threshold, suggested reorder qty, deficit, plus links to the item's detail page and stock-in form. New `nav-reorder` link in `base.html` for admin/manager/office. No mutations, no audit, no movement type, no new tables ‚Äî first read-only step in DoD #6's chain. PO2 will land the draft-PO write surface.
- **S5** ‚Äî Taxonomy: custom field defs per leaf node (Manager-owned). New `/admin/taxonomy/{node_id}/fields[?show=‚Ä¶]` list and `/new` form, `POST /admin/taxonomy/{node_id}/fields` create, plus flat `/admin/taxonomy/fields/{id}/edit|archive|unarchive` for the rest. Migration `0006_create_taxonomy_field_defs.py` adds the table with seven field types (text, number, decimal, date, boolean, select, multiselect), JSON `options_json`, `required` boolean, and partial-unique-by-name *and* by-key indexes scoped per node (across active + archived rows). Keys auto-derive from names (lowercase / underscored / alnum-only). Field defs only attach to *leaves*: a top-level node with no active sub-categories, or any sub-category. Creating a sub-category under a node with active field defs is now blocked (400) ‚Äî manager must archive the defs first. Audit vocabulary: `taxonomy_field_def.{created,updated,archived,unarchived}`.
- **S2** ‚Äî Locations CRUD (Manager-owned): new `/admin/locations` server-rendered list / new / edit / archive / unarchive routes behind `require_role(Role.MANAGER)`. Migration `0004_create_locations.py` adds the `locations` table (`name` unique across active + archived rows). Mirrors the supplier route shape ‚Äî same audit semantics (`entity_type="location"` + sparse update diffs) and same flash-after-mutation UX. The base layout gains a "Locations" nav link visible to Manager + Admin.
- **S1** ‚Äî Suppliers CRUD (Manager-owned): new `/admin/suppliers` server-rendered list / new / edit / archive / unarchive routes behind `require_role(Role.MANAGER)`. Migration `0003_create_suppliers.py` adds the `suppliers` table (`name` unique). Every mutation writes an `audit_log` row with `entity_type="supplier"` and a sparse before/after diff for updates. The base layout gains a one-shot flash region (`role="status"`) and a "Suppliers" nav link visible to Manager + Admin.
- **F3** ‚Äî Audit log infrastructure (append-only): new `audit_log` table (Alembic migration `0002`) + `record_audit()` helper in `app/audit.py` that flushes a row in the caller's transaction. Wired retroactively into `upsert_user_from_userinfo` (logs `user.created` and, for the bootstrap admin promotion, `user.bootstrap_admin_granted`) and the admin role/status mutations (`user.role_assigned`, `user.status_changed`). DB-level immutability triggers reject UPDATE/DELETE on `audit_log` for both SQLite and Postgres.
- **F2.1** ‚Äî Admin user-management UI: `/admin/users` is now an HTML page where admins can assign roles and change account status (pending ‚Üí active, active ‚Üí disabled). New POST routes `POST /admin/users/{id}/role` and `POST /admin/users/{id}/status` with server-side guards (admin can't demote or disable themselves, can't activate a roleless user). Pending users sort to the top of the list.
- **F2** ‚Äî Google SSO + users + roles: `users` table (Alembic migration), `Role` and `UserStatus` enums, Authlib OAuth login + callback, signed-cookie sessions, `require_role` dependency, role-gated `/admin/users`, anonymous/pending/welcome index page, dev/test-only login backdoor for Playwright. `SECRET_KEY` and Google credentials are now required when `APP_ENV=prod`.
- **F1** ‚Äî Project skeleton and verification harness: FastAPI app with `/health`, SQLAlchemy + Alembic wired (SQLite dev), pydantic-settings config, pytest unit + integration + Playwright e2e harness, `make check` green end-to-end.
