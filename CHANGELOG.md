# Changelog

High-level "what shipped" log. One line per slice. Newest at the top.

## Unreleased

- **MANPO1** — Manual PO creation (alternative to the reorder-dashboard auto-draft). New `+ New PO` button on `/admin/purchase-orders` opens `/admin/purchase-orders/new` — pick a supplier, fill in arbitrary lines (item + qty + optional expected unit cost), set expected date + notes, save as DRAFT. Same audit shape as the auto-draft path plus a ``source: "manual"`` marker. Manager + Office can create; same role gating as the reorder path. 17 new integration tests; RBAC sweep extended.
- **CSV-UPLOADS-SPEC** — Design-only `CSV uploads spec.md` covering bulk-upload for items / suppliers / locations / taxonomy. Mirrors the existing CSV download format; create-only in v1 (idempotent re-upload skips existing ids); dry-run preview default. Not yet implemented.
- **FDINH1** — Field-def inheritance through the taxonomy. Field defs on a top-level or sub-category now propagate to every descendant — defining a field once on "Rings" appears on every Silver / Gold / 925 sub-category's item form. Sub-categories can still add their own additive fields. ``_get_active_field_defs`` walks the ``parent_id`` chain and merges (root → leaf, with sort_order within each level). New tree-wide key-uniqueness check rejects a key that already exists on an ancestor or descendant (sibling-level collisions are still allowed). The legacy "fields can only be added to a leaf" gate is dropped; the legacy "cannot add sub-cat under a node with active field defs" gate is dropped (those fields now correctly inherit to new sub-cats). Field-defs list view renders an "Inherited from <ancestor>" section per ancestor with active fields. 12 new integration tests; 6 pre-existing leaf-only tests converted to assert the new inheritance behaviour.
- **POIT1** — Inbound PO in-transit visibility (Slice 3 of proposed scope-addition for stage tracking + in-transit transfers — **completes the scope addition**). New `POStatus.IN_TRANSIT` value sits between `SENT` and `PARTIALLY_RECEIVED`. New `purchase_orders.shipped_at` datetime column (migration `0020_po_in_transit`). New `POST /admin/purchase-orders/{po_id}/mark-shipped` (Manager+Office) flips `SENT` → `IN_TRANSIT`, records `shipped_at`, optional `expected_date` override; audit `purchase_order.shipped`. Receive flow accepts `IN_TRANSIT` alongside `SENT` and `PARTIALLY_RECEIVED` — marking-as-shipped is purely visibility, never a gate. Dashboard gains an "In-transit POs" stat card with earliest expected arrival; "Open POs" widget includes IN_TRANSIT in its count. PO detail template gains a "Mark as shipped" form on SENT POs and a "shipped on …" line afterwards. 12 new integration tests; RBAC sweep extended.
- **TR1** — Internal Transfer Orders (Slice 2 of proposed scope-addition for stage tracking + in-transit transfers). New `transfer_orders` + `transfer_order_lines` tables; new `transfer_order_id` FK on `stock_movements` to link the two TRANSFER movements (ship + receive) emitted per TO. Two-event lifecycle: DRAFT → SHIPPED (each line's item is validated to be at source, then its `location_id` becomes NULL = in transit) → RECEIVED (each line's item flips to the destination location). Full receipt only; discrepancies handled by separate adjustment movements. Cost engine is **never** invoked on TO movements. New module `app/transfers.py` mounted at `/admin/transfers` (list with status tabs + CSV export, create draft, detail with ship/receive/cancel buttons). Existing instant-flip `/admin/items/{id}/transfer` route preserved unchanged as a "quick relocate" path. Workshop+ can view; Office + Manager + Admin can create/ship/receive/cancel. Dashboard gains an "In-transit items" stat card linking to the SHIPPED filter. 18 new integration tests; RBAC sweep extended.
- **STG1** — Lifecycle stages (Slice 1 of proposed scope-addition for stage tracking + in-transit transfers). New `taxonomy_stages` table owned by top-level taxonomy nodes; `items.current_stage_id`; new `STAGE_CHANGE` movement type with `from_stage_id` / `to_stage_id` FKs on `stock_movements`. Manager-owned stages CRUD at `/admin/taxonomy/{node_id}/stages` (+ flat `/admin/taxonomy/stages/{id}/...` for edit/archive). Workshop+ stage-transition flow at `/admin/items/{item_id}/stage` writing append-only movements (qty=0) with required reason. Items list + CSV gain a `Stage` column; item detail shows the stage and a "Change stage" action. New items default to the category's `is_initial` stage. 26 new integration tests; RBAC sweep table extended for 9 new routes. No MISSION.md change — logged under "Proposed scope changes" in PROGRESS.md.
- **CI1** — GitHub Actions CI workflow with Postgres parity: new `.github/workflows/ci.yml` runs on push/PR to main. Single job on `ubuntu-latest` with Postgres 16 service container (pg_isready health check). Steps: checkout → astral-sh/setup-uv → uv sync → playwright install → `make lint` → `make typecheck` → `make test` (SQLite default) → `make test` with `TEST_DATABASE_URL=postgresql+psycopg://test:test@localhost/test_uc` (Postgres parity smoke) → `make e2e`. Backed by 16 new forcing-function tests in `tests/integration/test_ci_config.py`. **DoD #11 ticked — all 12/12 DoD items now complete.**
- **P4 + DOC9** — Fly.io deploy infrastructure + README deploy section: new `Dockerfile` (two-stage uv build), `.dockerignore`, `fly.toml` (Fly.io app config with `release_command = "alembic upgrade head"`). `Makefile` gains `make deploy` + `make deploy-logs`. `README.md` `## Deployment` section filled (prerequisites → first deploy → Postgres attach → secrets → BOOTSTRAP_ADMIN_EMAIL → email → ongoing); Quick Links deployed-URL and Tech Stack deploy-target `_TODO` blocks resolved. **DoD #12 ticked** — 11/12 DoDs complete.
- **OAUTH3** — Server-side OAuth stub + Playwright walk: new `app/oauth_test_stub.py` provides a minimal local OAuth 2.0 provider (GET `/authorize`, POST `/token`, GET `/userinfo`) mounted only when `APP_ENV=test` + `OAUTH_STUB_MODE=1`. Authlib's Google client is registered against stub endpoints in stub mode (no discovery, no JWT/id_token). New `oauth_stub_app_server` e2e fixture boots a separate uvicorn with the stub active. Two new Playwright walks exercise the real `/auth/google/login` + `/auth/google/callback` routes end-to-end in a browser. **DoD #1 + DoD #10 ticked** — 10/12 DoDs complete.
- **R5i** — Admin users CSV export: `GET /admin/users?format=csv` exports the admin user-management list as a downloadable CSV (`users.csv`), gated to Admin only same as the HTML view. Six columns: `id`, `email`, `name`, `role`, `status`, `created_at`. Enums pre-coerce to their `.value` strings (e.g. `"manager"`, `"active"`); pending users with no role render an empty `role` cell; `created_at` is preserved as a full ISO datetime. Template gains an `admin-users-csv-link` "Download CSV" link above the users table. Tenth CSV-export surface; same `csv_branch` pattern as the nine prior list views.
- **AC1** — Audit-view CSV export: `GET /admin/audit?format=csv` exports every audit-log row as a downloadable CSV (`audit_log.csv`), gated to Manager + Admin same as the HTML view. Eight columns: `id`, `created_at`, `actor_email`, `action`, `entity_type`, `entity_id`, `before_json`, `after_json`. JSON cells render as compact JSON; system actions render with empty `actor_email`. The CSV branch ignores HTML pagination — exports are snapshots, not windows. Template gains an `audit-csv-link` "Download CSV" link. Eighth CSV-export surface; same `csv_branch` pattern as the seven prior list views.
- **C1** — Per-item `requires_checkout` flag UI + checkouts schema foundation: new `Checkout` ORM model in `app/models.py` mirroring MISSION §6 columns (no status enum — open vs returned derived from `returned_at IS NULL`). Migration `0013_create_checkouts.py` creates the table + four indexes (per item / item_unit / user / returned_at). Items list gains a "Checkout?" column showing "Yes" / "—" plus a new filter nav (`All items` / `Requires checkout`) backed by `?requires_checkout=yes` (silent coerce on any other value). The items form gains an explanatory help note under the existing checkbox. No new routes — check-out / check-in routes land in C2 / C3, manager "currently out / overdue" view in C4. C1's job is to land the foundation so those slices don't have to also ship a migration.
- **PO4** — Email PO PDF to supplier (draft → sent): new `POST /admin/purchase-orders/{po_id}/send` (Manager + Office). Validates draft + active supplier with non-blank email, renders the PDF via `app.pdf.render_po_pdf`, builds an `EmailMessage` with HTML body + PDF attachment, delivers via the configured backend, then flips `po.status = SENT` + `po.sent_at = now()` and writes a `purchase_order.sent` audit row. New `app/email_backend.py` module exposes `EmailMessage` / `EmailAttachment` / `EmailBackend` Protocol + `ConsoleEmailBackend` (in-memory outbox + stdout summary; default in dev/tests) and `SmtpEmailBackend` (stdlib `smtplib` + `email.message`). Switch via the `EMAIL_BACKEND` env var. Detail page gains `po-send-form` on drafts with supplier email, or `po-send-blocked-no-email` otherwise.
- **PO2** — Draft PO from a supplier group: new module `app/purchase_orders.py` with three routes — `POST /admin/reorder/draft-po` (Manager + Office; creates a draft PO from a supplier group on the reorder dashboard), `GET /admin/purchase-orders` (list, status-filterable), `GET /admin/purchase-orders/{po_id}` (read-only detail). New tables `purchase_orders` + `purchase_order_lines` (migration `0011`); new `POStatus` enum (only `draft` written in PO2). New `nav-pos` link in `base.html` for admin/manager/office. The reorder dashboard gains a "Draft PO" button per active-supplier group; archived-supplier and "(no supplier)" groups render distinct blocked-notes. Lines auto-default `qty_ordered` from `item.reorder_qty` (or deficit, or 1) and `expected_unit_cost` from the most recent cost layer (or null). Audit: `purchase_order.created` carrying the full lines snapshot. Editing per-line fields and cancelling drafts is deferred to PO2b — without it MISSION §3's "expected unit cost is editable" requirement is not yet satisfied, so DoD #6 cannot tick.
- **PO1** — Reorder dashboard: new module `app/reorder.py` + `GET /admin/reorder` for **Manager + Office** (read-only; Workshop + pending blocked). Surfaces items where `current_qty <= reorder_threshold AND archived_at IS NULL`, grouped by supplier (with a synthetic "(no supplier)" bucket and an "(archived)" suffix on archived suppliers). Each row shows current qty, threshold, suggested reorder qty, deficit, plus links to the item's detail page and stock-in form. New `nav-reorder` link in `base.html` for admin/manager/office. No mutations, no audit, no movement type, no new tables — first read-only step in DoD #6's chain. PO2 will land the draft-PO write surface.
- **S5** — Taxonomy: custom field defs per leaf node (Manager-owned). New `/admin/taxonomy/{node_id}/fields[?show=…]` list and `/new` form, `POST /admin/taxonomy/{node_id}/fields` create, plus flat `/admin/taxonomy/fields/{id}/edit|archive|unarchive` for the rest. Migration `0006_create_taxonomy_field_defs.py` adds the table with seven field types (text, number, decimal, date, boolean, select, multiselect), JSON `options_json`, `required` boolean, and partial-unique-by-name *and* by-key indexes scoped per node (across active + archived rows). Keys auto-derive from names (lowercase / underscored / alnum-only). Field defs only attach to *leaves*: a top-level node with no active sub-categories, or any sub-category. Creating a sub-category under a node with active field defs is now blocked (400) — manager must archive the defs first. Audit vocabulary: `taxonomy_field_def.{created,updated,archived,unarchived}`.
- **S2** — Locations CRUD (Manager-owned): new `/admin/locations` server-rendered list / new / edit / archive / unarchive routes behind `require_role(Role.MANAGER)`. Migration `0004_create_locations.py` adds the `locations` table (`name` unique across active + archived rows). Mirrors the supplier route shape — same audit semantics (`entity_type="location"` + sparse update diffs) and same flash-after-mutation UX. The base layout gains a "Locations" nav link visible to Manager + Admin.
- **S1** — Suppliers CRUD (Manager-owned): new `/admin/suppliers` server-rendered list / new / edit / archive / unarchive routes behind `require_role(Role.MANAGER)`. Migration `0003_create_suppliers.py` adds the `suppliers` table (`name` unique). Every mutation writes an `audit_log` row with `entity_type="supplier"` and a sparse before/after diff for updates. The base layout gains a one-shot flash region (`role="status"`) and a "Suppliers" nav link visible to Manager + Admin.
- **F3** — Audit log infrastructure (append-only): new `audit_log` table (Alembic migration `0002`) + `record_audit()` helper in `app/audit.py` that flushes a row in the caller's transaction. Wired retroactively into `upsert_user_from_userinfo` (logs `user.created` and, for the bootstrap admin promotion, `user.bootstrap_admin_granted`) and the admin role/status mutations (`user.role_assigned`, `user.status_changed`). DB-level immutability triggers reject UPDATE/DELETE on `audit_log` for both SQLite and Postgres.
- **F2.1** — Admin user-management UI: `/admin/users` is now an HTML page where admins can assign roles and change account status (pending → active, active → disabled). New POST routes `POST /admin/users/{id}/role` and `POST /admin/users/{id}/status` with server-side guards (admin can't demote or disable themselves, can't activate a roleless user). Pending users sort to the top of the list.
- **F2** — Google SSO + users + roles: `users` table (Alembic migration), `Role` and `UserStatus` enums, Authlib OAuth login + callback, signed-cookie sessions, `require_role` dependency, role-gated `/admin/users`, anonymous/pending/welcome index page, dev/test-only login backdoor for Playwright. `SECRET_KEY` and Google credentials are now required when `APP_ENV=prod`.
- **F1** — Project skeleton and verification harness: FastAPI app with `/health`, SQLAlchemy + Alembic wired (SQLite dev), pydantic-settings config, pytest unit + integration + Playwright e2e harness, `make check` green end-to-end.
